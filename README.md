## 项目简介

HongJie 是一个专为**中小型网站**设计的**轻量化多平台认证网关**，将**反爬虫、防扫号、用户认证、登录体系**四大功能集于一体，**可自由扩展、无缝接入新老项目**，开箱即用。

### 🎯 解决的核心痛点

| 痛点 | 传统方案 | HongJie 方案 |
|------|----------|--------------|
| **用户注册登录** | 需要开发手机号/邮箱验证、密码加密、找回密码、Session/JWT 管理等全套系统，动辄数百行代码 | **零开发**，扫码即登录，天然免密码 |
| **用户数据存储** | 必须存储手机号、密码哈希、盐值等敏感信息，存在数据泄露风险 | **零存储**，只记录 OpenID，无密码可泄露 |
| **GDPR/个保法合规** | 需要处理数据删除请求、隐私协议、用户同意记录等复杂流程 | **天然合规**，根本不存数据 |
| **爬虫/刷接口防护** | 需要配置 WAF、验证码、限流规则，维护成本高 | **多层防御内置**，PoW + Turnstile + 限流开箱即用 |
| **扫号攻击防护** | 需要开发登录失败锁定、IP 封禁、行为分析等 | **内置动态封禁**，暴力尝试自动拉黑 |
| **新老项目接入** | 需要改造现有认证逻辑，成本高、风险大 | **透明代理**，一行代码改动即可接入 |
| **多平台用户覆盖** | 需要对接多个 OAuth 平台，重复开发 | **插件化设计**，一次架构支持多平台扩展 |

### 🚀 无缝接入新老项目：透明代理设计

**对后端完全透明，现有代码只需改动 1 行：**

```python
# 你的老项目（PHP/TP5/Laravel/ThinkPHP）
# 原来：从 session 获取用户
$user_id = $_SESSION['user_id'];

# 现在：从 Header 获取（网关自动注入，不限平台）
$user_id = $_SERVER['HTTP_X_USER_ID'];
$platform = $_SERVER['HTTP_X_PLATFORM'];  # wechat/alipay/douyin/kuaishou
```

```
┌─────────────────────────────────────────────────────────────────┐
│                    老项目接入流程（5 分钟）                       │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  步骤 1：启动 HongJie 网关                                      │
│  └─ $ docker-compose up -d                                     │
│                                                                 │
│  步骤 2：修改老项目 Nginx/Apache 配置                            │
│  └─ 将请求代理到网关（或修改 DNS 指向网关）                      │
│                                                                 │
│  步骤 3：修改老项目代码（1 行）                                  │
│  └─ session('user_id') → request.header('X-User-ID')          │
│                                                                 │
│  步骤 4：部署小程序（微信/支付宝/抖音/快手）                      │
│  └─ 根据目标用户群体选择部署平台                                 │
│                                                                 │
│  完成！原有业务逻辑完全不受影响 ✅                               │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**支持接入的项目类型：**

| 项目类型 | 接入方式 | 改动成本 |
|----------|----------|----------|
| **PHP 老项目** (ThinkPHP/Laravel/原生) | 修改 `$_SESSION` 为 `$_SERVER['HTTP_X_USER_ID']` | 1 行 |
| **Go 项目** (Gin/Echo/原生) | 修改 `c.Get("user_id")` 为 `c.Request.Header.Get("X-User-ID")` | 1 行 |
| **Node.js 项目** (Express/Koa) | 修改 `req.session.userId` 为 `req.headers['x-user-id']` | 1 行 |
| **Python 项目** (Flask/Django) | 修改 `session.get('user_id')` 为 `request.headers.get('X-User-ID')` | 1 行 |
| **Java 项目** (Spring Boot) | 修改 `@SessionAttribute` 为 `@RequestHeader` | 1 行 |
| **Ruby on Rails** | 修改 `session[:user_id]` 为 `request.headers['HTTP_X_USER_ID']` | 1 行 |
| **纯静态网站** | 无需改动，通过 JS 读取 Header（需后端配合） | 0 行 |

## 安全机制详解

### 第一层：IP 动态封禁（防扫号）
- 根据违规次数自动升级惩罚：5次→10分钟，10次→1小时，20次→24小时
- 定期清理过期封禁记录，无需人工干预
- **效果**：暴力破解密码？第 6 次尝试就被封 10 分钟

### 第二层：PoW 工作量证明（反爬虫）
- 每次二维码请求前需完成 SHA-256 计算，难度 16 bit
- 浏览器端平均需要计算数千次哈希（约 0.5-2 秒）
- **效果**：爬虫成本提高 1000 倍，普通用户无感知

### 第三层：Cloudflare Turnstile（防机器人）
- 可选的无感人机验证，无需用户点击
- **效果**：自动识别 headless 浏览器、自动化脚本

### 第四层：设备指纹限流（防刷接口）
- 基于 IP + 设备指纹的组合限流，60 秒冷却
- 浏览器 Canvas 指纹 + 屏幕特征 + 时区等
- **效果**：同设备/IP 高频请求自动限流

### 第五层：一次性动态链路（防重放）
- 每个二维码携带唯一令牌（32 位 hex）
- 60 秒有效期，使用即失效，无法二次使用
- **效果**：二维码截图分享无效，防止中间人重放

## 常见问题

### Q: 如何更新微信 access_token？
A: 网关会自动定时刷新（提前 300 秒），无需手动干预。

### Q: 小程序码的有效期是多久？
A: 二维码本身是各平台生成的长期有效码，但对应的 `scene_token` 有效期为 60 秒，超时需重新获取。

### Q: 如何自定义静态文件？
A: 修改 `static_files.json` 添加路径映射，文件需放在容器内对应路径。

### Q: 后端服务如何获取用户 OpenID？
A: 网关会在代理请求中添加 `X-Open-ID` Header。

### Q: 不使用 Docker 可以部署吗？
A: 可以，直接 `go run main.go` 即可，需要确保 cert.pem 和 key.pem 路径正确。

### Q: 如何扩展支持支付宝/抖音/快手？
A: 网关采用插件化设计，只需实现对应平台的 `AuthPlugin` 接口：
   - 实现 `GetUserInfo(code string)` 获取用户信息
   - 实现 `GenerateQRCode(scene string)` 生成小程序码
   - 复用现有的场景管理和令牌交换逻辑
   - 欢迎提交 PR 贡献代码

### Q: 多平台如何区分用户来源？
A: 网关会在代理转发的 Header 中添加 `X-Platform` 字段，后端可通过该字段识别用户来自哪个平台。

### Q: 网关如何水平扩展？
A: 网关无状态设计，可部署多实例 + 负载均衡，会话信息存储在客户端 Cookie 中。

### Q: 示例网站 https://zhiliaoya.cn/ 稳定吗？
A: 该站点用于功能演示和技术验证，不保证 100% 可用性，建议自行部署测试。
